2. Impact of POI Fast-Travel + 8-Way Numpad Drift on Architecture
The paradigm shift is significant but reduces complexity in the right places. Key consequences:
2.1 The grid stops being the player's navigation surface
Under the old model, grid.js had to support per-tile boat traversal, neighbor walking, and continuous momentum across hundreds of cells. Under D-001/D-002:
The **macro layer** is a graph of POIs (nodes) with edge costs (time, fuel, noise). This is not a grid problem — it's a small weighted graph.
The **micro layer** is a bounded 8-way drift space *inside* a POI cell. Momentum, wind, and station-keeping live here, operating on a tiny local frame (effectively a 3×3 to ~9×9 sub-region), not the whole lake.
Result: grid.js becomes a terrain query service (depth, bottom, cover at coordinates) consumed by casting, sonar, and the structure index — not by the player's movement loop.
2.2 New modules required, some old ones repurposed
**New:** `world/poiGraph.js` — nodes, edges, fast-travel cost calculator.
**New:** `navigation/poiTravel.js` — executes a fast-travel jump (consumes fuel/time, emits arrival event).
**New:** `navigation/microDrift.js` — the 9-direction (8 + center/hold) local frame, wind/current vectors applied per tick, momentum accumulator scoped to the current POI.
**New:** `navigation/stationKeeping.js` — implements D-003 (Active Oars vs Passive Trolling Motor), pure function over current drift vector.
**Repurposed:** `navigation/boatController.js` becomes a thin **mode router** ( `TRAVELING` | `AT_POI` ) that delegates to `poiTravel` or `microDrift`. The old "intent → motor → wind → momentum → override → commit" pipeline now lives entirely inside `microDrift.js` against a local frame.
**Repurposed:** `motor.js` splits responsibility — outboard logic is consumed by `poiTravel`; trolling motor logic is consumed by `stationKeeping`.
**Demoted:** `shallowOverride.js` is no longer a player-safety override (the player can't accidentally ram land — they fast-travel to known POIs). It survives only as a validator inside `poiTravel` to reject travel routes that pass through impassable shallows when relevant, and inside `microDrift` to clamp drift at POI boundaries.
2.3 Data model changes
**POI record:** `{ id, name, coords, type (point/dropoff/weedbed/timber/dock), traits, fuelCostFromDock, discovered }`. POIs are emitted by `lakeGenerator.js` and become the canonical entries in `structureIndex.js`. **The structure index and the POI graph are the same dataset viewed two ways** — AI sees it as queryable structure; the player sees it as a fast-travel menu. This is a major win: one source of truth for both player navigation and AI targeting.
**Local frame state:** `{ poiId, offset:{dx,dy} in [-R..R], driftVector, momentumStreak, anchored:bool }`.
**Discovery flag:** POIs may be undiscovered until revealed by sonar sweep, tournament intel, or scripted intro. This becomes a real gameplay axis without re-introducing blind exploration.
2.4 What this removes from the engine
Per-tile pathfinding for the player. Gone.
Per-tick boat-vs-grid collision checks across the whole lake. Gone.
The compounding "long drift across many tiles" failure mode in H-001 — drift is now bounded by the POI's local frame radius.
H-001 is not eliminated; it's localized to microDrift.js, which is exactly where we want it.
3. Open Question Resolutions
OQ-1 — Turn Model: Strict turn-based with player-paced sub-phases.
POI fast-travel is intrinsically discrete (you arrive, time advances by computed cost).
Micro-drift is naturally per-turn (each numpad press = one tick of drift resolution).
The 4-part cast (D-004) and the fight loop are timing-based **within** a turn; they use real-time input windows but do not advance the world clock until resolved.
AI scheduling under D-005 is dramatically simpler against a turn boundary — each AI gets one `observe → decide → act` pass per world tick, deterministic under the seeded RNG (§5d).
Recommendation: turn-based world clock; embedded real-time input windows for cast timing and fight tension. This is the only model that preserves §5d replayability while honoring D-004.
OQ-2 — Coordinate System: Sparse `Map<"x,y", tile>` for the lake; integer offset coords for micro-drift.
The lake is non-rectangular and most of the grid will be empty/land — sparse is the right primitive and cheaper to serialize for replays.
The player never traverses raw tiles anymore (per §2.1), so we don't need axial-grid neighbor math for movement — we need fast point lookups for casting/sonar/structure tagging. `Map` keyed by `"x,y"` is optimal for that workload.
**Micro-drift uses a separate, small offset coordinate** (e.g., `dx,dy ∈ [-R..R]` around the POI center). This is a different coordinate space and should not share types with lake coords — keep them in distinct modules to prevent drift between layers.
Axial/hex was attractive only when continuous movement mattered. Under D-001, it doesn't.
Recommendation: Map&lt;string, Tile&gt; in world/grid.js; plain {dx,dy} integer offsets in navigation/microDrift.js. Single coordKey(x,y) helper to enforce key format.
OQ-3 — Input Layer: Abstracted input adapter, not raw keypress timestamps.
Accessibility (§5c spirit) demands the engine support keyboard, switch input, screen-reader voice commands, and potentially game-controller mapping without engine changes.
The 4-part cast (D-004) needs millisecond-resolution release timing, but that requirement belongs to the **adapter**, which normalizes any input source into a stream of `{type, timestamp, payload}` events on the bus.
Keeps the engine deterministic: replays feed recorded input events through the adapter, identical to live play.
Recommendation: a new core/inputAdapter.js that publishes INPUT_* events on eventBus. Cast timing, numpad drift, and menu navigation all consume the same normalized stream. No engine module ever reads keydown directly.
