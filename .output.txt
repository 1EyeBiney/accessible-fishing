1. Critical Evaluation вЂ” Where Bottlenecks Will Emerge
1.1 Grid + Momentum coupling
The grid is non-rectangular, and momentum is a per-turn accumulator influenced by wind vectors, motor input, and the 3-tile shallow-water override. Three real risks:
**Vector resolution order ambiguity.** If wind, drift momentum, motor thrust, and the shallow-water cutoff are resolved in inconsistent order, the same inputs will produce different boat positions across turns. This is the #1 source of "logic drift" in headless sims. Resolution order must be a single, declared pipeline (e.g., `intent в†’ motor в†’ wind в†’ momentum accumulator в†’ collision/override в†’ commit`).
**Non-rectangular grid edge cases.** Sparse arrays or jagged rows make neighbor lookups (`N/S/E/W` + diagonals) error-prone. A single `isWater(x,y)` predicate must be the only authority вЂ” never inline checks.
**Override interaction with momentum.** When the outboard auto-cuts at 3 tiles, what happens to accumulated momentum? Carry it, halve it, or zero it? This must be a declared rule, not emergent behavior.
1.2 AI competitor logic at tournament scale
A leaderboard of simulated anglers each running a "brain mode" against the same grid/clock is the second bottleneck:
**Per-tick cost.** If every AI walks the grid each tick to evaluate structure targets, cost is `O(AI_count Г— grid_cells Г— candidate_lures)`. With 10вЂ“20 opponents, this stalls a turn-based loop and destroys any future real-time mode.
**Solution direction:** AI should not see the grid directly. They should query a **structure index** (precomputed: drop-offs, weed beds, timber clusters tagged with coordinates and trait scores). Brain modes operate on this index, not raw tiles.
**Determinism.** Tournaments require reproducible runs (replays, leaderboard fairness). All randomness вЂ” weather, AI dice rolls, fish strike checks вЂ” must flow through a single seeded RNG service. Multiple `Math.random()` calls scattered across modules will make debugging impossible.
1.3 Cross-cutting concerns
**Time/turn authority.** Wind shifts, lure decay, bait perishing, AI actions, and fish behavior all tick. One clock service, or you get desync between "it's 2pm for the player but bait is still fresh from 9am."
**Audio event bus.** Headless в‰  silent. Every state change worth announcing (cast release, sonar ping, drift warning, line tension) must emit a structured event, not a `console.log`. The audio layer subscribes; the engine never calls audio directly.
**Cast pipeline crossing modules.** The 4-part cast touches equipment (power), input timing (release), grid (target validation), and physics (entry). This is the most likely place for a god-function to creep in. It must be a thin orchestrator over single-purpose modules.

Why this split holds up under the Watchdog rules:
Each file has one job вЂ” no file owns both state and logic.
`boatController.js` and `castResolver.js` are the only orchestrators; everything else is pure or near-pure.
AI never imports `grid.js` вЂ” only `structureIndex.js`. This enforces the perf boundary at the import level.
Audio is decoupled by `eventBus.js`. The engine remains genuinely headless and testable.
3. Development Roadmap
Phased so each step is independently verifiable in a headless test harness before the next begins.
Phase 0 вЂ” Foundations
`core/eventBus.js`, `core/rng.js`, `core/clock.js`, `core/stateStore.js`.

`world/tileTraits.js` вЂ” enums and trait shapes only.

`world/structureIndex.js` вЂ” build once after generation; expose query API.

`navigation/wind.js` and `navigation/momentum.js` as pure functions.

`navigation/shallowOverride.js` вЂ” pure, given grid + pos + intended vector.
`navigation/boatController.js` вЂ” declared resolution pipeline; fixture-based tests covering: pure drift, motor vs wind, momentum carry, override at 3-tile boundary, override interaction with momentum.

`equipment/rods.js`, `equipment/lures.js`, `equipment/liveBait.js`, `equipment/durability.js`.

Phase 4 вЂ” Casting (Pasture adaptation)

Phase 5 вЂ” Fish & strikes
