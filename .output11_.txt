1. Water Tile Schema — Locked
Phase 1 tile schema is now closed for v0.1 with these rulings folded in (will be lifted into the brief at the v1.6 bump):

Schema impacts of the new rulings
**Flow moves off the tile.** `traits.flow` is removed from the tile schema (per ruling 5). Per-POI flow lives on the POI / structure-index entry as a single vector. Tiles get smaller and uniform-flow logic gets cheaper.
**Frame-boundary penalty (ruling 5) is a navigation concern, not a tile concern.** The tile schema is unchanged by it. The penalty hooks into `navigation/microDrift.js`: when the integrated drift offset would exceed `frameRadius`, the boat snaps to a "repositioning" sub-state that consumes a configurable time penalty on the tournament clock and re-anchors at the POI center. This sits naturally beside D-013 (time-as-penalty) and reuses `clock.schedule()` — no new mechanism needed.
**Replay snapshot scope (ruling 8):** serializer whitelist is `id`, `coord`, `traits`, `state.spook`, `state.occupancy`. `state.events[]` is dropped at the boundary.
2. Fish Finder Pivot — Targeting Data Flow
This is the right call. Manual cursor traversal of an 81-tile micro-frame is hostile to screen-reader users; the Fish Finder turns the same gameplay surface into a discoverable menu, which is exactly the navigation pattern POI fast-travel established at the macro layer (D-001). The pivot is consistent: the player navigates menus of pre-classified targets at every scale, never raw geometry.
2.1 What gets removed
The cursor-on-micro-frame targeting model from earlier Phase 4 discussion (D-011 stays, but the way the player commits a target changes).
Any direct use of arrow keys to traverse the micro-grid for casting purposes.
2.2 What gets added
A new module: `equipment/fishFinder.js` already existed in §9 — it now becomes substantially more central. It owns scan execution, return formatting per finder tier, and selection state.
A new module: `casting/targetSelector.js` — the menu-driven target-lock surface. Pure FSM consuming `INPUT_*` events, mutating only its own slice of state, emitting a single `TARGET_LOCKED` event the cast pipeline subscribes to.
A new tile state field: `state.pressure` (see §3).

2.6 The "presenceHint" abstraction — accessibility safety valve
Even at PRO/ELITE, never reveal exact fish-by-tile counts in the menu — it would turn the game into an inventory exercise. Instead, return categorical hints: NONE | TRACE | SCATTERED | SCHOOLED. This protects strategic ambiguity (the player still has to make a judgment call) while giving real progression value.
3. The Fished-Out Mechanic — state.pressure
3.1 Conclusion: separate variable, separate decay rate, identical compute-on-read pattern.
Folding fished-out into spook is tempting but wrong. They're orthogonal:
Axis Spook Pressure
Cause Acoustic disturbance (loud splash, motor) Successful angling activity (casts on this tile)
Resets via Time decay (12s/level) Time decay (much slower) AND fresh Fish Finder scan that re-evaluates the area
Affects Strike probability (short-term suppression) Fish Finder candidate ranking + strike probability (long-term suppression)
Player remedy Wait quietly, or move Move boat to a new POI / re-scan after pressure decays
Replay determinism Critical — drives strikes Critical — drives finder rankings
Conflating them would force a single decay rate that is wrong for both behaviors and would couple the cast pipeline to the finder pipeline, violating single-purpose modules.
3.2 Schema addition (final; closes the v0.1 tile schema)
```
"state": {
  "spook": { /* unchanged */ },
  "pressure": {
    "level":          0,         // 0..MAX_PRESSURE (recommended 5)
    "updatedAtMs":    0,         // tournament clock at last write
    "lastCastAtMs":   0,         // for diagnostics + finder freshness
    "lastCatchAtMs":  0          // for finder freshness display
  },
  "occupancy":   { /* unchanged */ }
}
```
3.3 Increment table (proposed — open for ruling)
Event Pressure increment
Cast lands on tile (any outcome) +1
Successful hookset on tile +1 (additional, on top of the cast)
Fish landed from tile +1 (additional, on top of hookset)
A tile fished hard and producing a catch jumps to +3 in one cycle — exactly the "spot is burned, move on" feel you described. A pure miss is +1.
3.4 Decay (proposed — open for ruling)
```
MAX_PRESSURE                = 5
PRESSURE_DECAY_MS_PER_LEVEL = 90,000   // 90s per level → ~7.5min full decay
```
Pressure decays much slower than spook. Spook is "the fish heard you for a moment;" pressure is "the school has wised up to this spot." This asymmetry is what makes the upgrade ladder matter — better boats get to more POIs faster, before pressure decays everywhere are reset.
3.5 Compute-on-read (same pattern as spook — H-001-aligned)
```
readPressure(tile, atMs):
    if tile.state.pressure.level == 0: return 0
    decayed = floor((atMs - tile.state.pressure.updatedAtMs) / PRESSURE_DECAY_MS_PER_LEVEL)
    return max(0, tile.state.pressure.level - decayed)
bumpPressure(tile, source, atMs):   // source: 'CAST' | 'HOOKSET' | 'CATCH'
    inc = source == 'CATCH' ? 1 : source == 'HOOKSET' ? 1 : 1
    current = readPressure(tile, atMs)
    tile.state.pressure.level       = min(MAX_PRESSURE, current + inc)
    tile.state.pressure.updatedAtMs = atMs
    if source == 'CAST':    tile.state.pressure.lastCastAtMs   = atMs
    if source == 'CATCH':   tile.state.pressure.lastCatchAtMs  = atMs
```
Module ownership: a new tiny module fish/pressureModel.js mirroring casting/castSpookModel.js. Both are pure compute-on-read services consumed by the finder, the strike model, and the resolver. No bus traffic per tile.
3.6 How it integrates with the Fish Finder
When the finder scans, each candidate's score is multiplied by (1 - pressure/MAX_PRESSURE) × (1 - spook/MAX_SPOOK). A fully-pressured tile drops out of the candidate list entirely (or appears with a "FISHED OUT" label depending on finder tier). This is what produces your "forces another scan + boat move" gameplay loop without any new tracking variable beyond the field added above.
3.7 Strike model integration
fish/strikeModel.js already consults castSpookModel.isSpooked. Add an analogous consult of pressureModel.readPressure:
```
strikeProbability =
   baseProb(species, lure, depth, time)
   × (1 - spook/MAX_SPOOK)
   × (1 - pressure/MAX_PRESSURE × PRESSURE_STRIKE_PENALTY)
```
PRESSURE_STRIKE_PENALTY (proposed 0.6) means even max pressure doesn't fully zero the chance — there's always a slim shot at a stubborn fish, which preserves the "Bill the Legend pulls one out of nowhere" narrative beat (D-006).
3.8 Determinism and storage cost
Pure compute-on-read: zero per-tick traffic, zero replay drift.
Two integers + two timestamps per fished tile. Sparse Map storage means non-fished tiles cost zero. For a tournament with ~200 cast events, peak storage is ~200 × 32 bytes = ~6.4 KB. Negligible.
Pressure is in the replay snapshot whitelist (it influences strikes and finder output). `lastCastAtMs` / `lastCatchAtMs` are diagnostic but cheap; recommend including them in the snapshot for replay-tool features (heatmap diffs, post-tournament analysis) — they don't affect determinism.
