Hub World & Economy Loop — Architectural Evaluation
STRICT NO CODE LOCK acknowledged. Brief v1.3 directive §0 honored. Phase 0 code from a prior session is recognized as purged per S-004; this response emits only architectural analysis and proposed brief edits for your review. No .js, .json, or script files will be created or modified.
1. State Management — HUB_MODE ↔ TOURNAMENT_MODE Routing
The cleanest discipline here is to treat game mode as a first-class state machine in `stateStore` and have engine.js act purely as a router that mounts/unmounts subsystems based on the current mode. No subsystem decides whether it should run; the engine decides what runs.
Proposed mode FSM
```
BOOT → FOCUS_TRAP → PROFILE_SELECT → HUB → (TOURNAMENT_BRIEFING) → TOURNAMENT_ACTIVE → TOURNAMENT_RESULTS → HUB
                                       ↘                                                ↗
                                        (LEADERBOARDS / SOUND_MENU / SHOPS / WORKSHOP)
```
`BOOT`: TTS engine warm-up, asset/manifest load. Bus only.
`FOCUS_TRAP`: holds screen-reader focus on a single live region until the user dismisses with a confirmed input. This is the audio analog of a splash screen.
`PROFILE_SELECT`: load/create persistent profile (see §1.4 below).
`HUB`: clock paused; only menu-tier subsystems mounted (shops, workshop, leaderboards, sound).
`TOURNAMENT_ACTIVE`: clock running; navigation/casting/AI/fish all mounted.
Key state-routing rules
**Single source of truth.** `state.mode` lives in `stateStore`. Every subsystem queries it instead of holding its own boolean. Mode transitions are centralised in a new helper, `core/modeRouter.js`, that performs `stateStore.mutate({mode: newMode})` plus emits `MODE_CHANGED` plus calls a registered enter/exit hook per subsystem.
**Clock is mode-aware, but mode does not control the clock directly.** `modeRouter` calls `clock.pause()` on enter-`HUB` and `clock.resume()` (or `clock.reset()` for a fresh tournament) on enter-`TOURNAMENT_ACTIVE`. This preserves §5e (single time authority) — only the router talks to the clock for mode reasons.
**Subsystem registration, not subsystem polling.** `engine.js` registers subsystems with a manifest like `{ id, mountInModes: ['TOURNAMENT_ACTIVE'], onMount, onUnmount }`. Mounting wires bus subscriptions; unmounting tears them down. This prevents the dangerous failure mode where, for example, the AI tournament scheduler keeps ticking in HUB because its `every(...)` handle was never cancelled. Hazard candidate (see §4 below).
**State partition.** State should be partitioned into `hub`, `tournament`, and `profile` slices with clear ownership, so a mode switch does not require a deep clone or reset of unrelated data:
- `state.profile` — persistent across runs; serialized to disk.
- `state.hub` — economy snapshot the player carries between tournaments (wallet, owned boat, owned tackle, inventory).
- `state.tournament` — disposable per-event run state (clock elapsed, current POI, micro-offset, cast phase, spook tiles, AI roster state, current standings).
- `state.session` — UI/menu cursor positions, focus state. Never persisted.
**Save points are mode boundaries.** Persist on enter-HUB after a tournament completes (winnings applied, gear unchanged). Never auto-save mid-tournament (it would let players quit-and-reload to dodge bad weather). This is a future Decision worth declaring.
Why engine.js does not grow logic
Per §9, engine.js is "tick loop, mode routing, no game logic." Adding a hub does not change that. Mode routing is delegated to core/modeRouter.js so engine.js remains thin. engine.js:
Owns the boot sequence (`BOOT → FOCUS_TRAP → PROFILE_SELECT → HUB`).
Owns the clock instance.
Drives `modeRouter` based on bus events (`REQUEST_ENTER_TOURNAMENT`, `REQUEST_RETURN_TO_HUB`, `TOURNAMENT_COMPLETED`).
2. Audio Decoupling — audio.js vs eventBus
§5f is non-negotiable: the engine never calls audio directly. The new audio.js is therefore a consumer of the bus, never a callee of the engine. Concrete contract:
Channel Direction Examples
Engine → Bus emit only CAST_LANDED, SONAR_PING, WIND_SHIFTED, INPUT_BLOCKED, MODE_CHANGED, CLOCK_RESUMED
Bus → Audio subscribe only audio.js subscribes via wildcard &#39;*&#39; and a routing table
Audio internal self TTS queue, music ducking, SFX mixer, accessibility mute groups
Module boundaries inside audio/
The §9 entry should be a directory, not a single file, to honour the modular-split rule:
`audio/audioManager.js` — boots TTS engine, mounts bus subscriptions, owns mute groups, owns ducking policy.
`audio/audioRoutes.js` — pure mapping table: `{ eventType → { channel, priority, voiceTemplate } }`. This is the only place that knows "event X means say Y." Swapping voices/skins = edit this file only.
`audio/ttsQueue.js` — priority queue with barge-in rules (e.g., `INPUT_BLOCKED` interrupts ambient narration).
`audio/sfxBank.js` — sample lookup; deterministic pitch variation via `rng.stream('audio')` so replays sound identical.
`audio/musicBed.js` — hub vs tournament beds, dynamic intensity ramp during fight.
Three things that must NOT happen
The engine must never `import audio from '...'`. Compliance check: `audio` should appear in zero `import` statements outside `engine.js`'s top-level boot wiring (and even there only to call `audioManager.boot()`).
`audio.js` must not write to `stateStore`. Audio is a sink, not a source. Exception: it MAY emit `AUDIO_QUEUE_DRAINED` style events back onto the bus for tests, but never mutate state.
TTS calls must never block the engine tick. The queue is async; the engine fires-and-forgets.
Determinism note (Hazard relation)
Audio output is observably non-deterministic across machines (TTS rate, sample variation). Replay determinism (H-004) only covers engine state — audio is a presentation layer. This boundary should be declared explicitly so future "replay diff" tools don't try to compare audio.
3. POI Travel Math — Boat Upgrades into poiTravel.js
A flat speed multiplier is too coarse and will trivialise the upgrade ladder. The boat is also noise, fuel, stability, and shallow access — not just speed. Recommend modelling the boat as a stat block consumed by poiTravel.js:
```
Boat stat block (proposed, conceptual):
  speedTilesPerMin       // base traverse rate
  shallowDraftMin        // minimum water depth the boat can enter
  noiseProfile           // 0..1, affects spook radius along travel path
  fuelType               // 'NONE' (oars) | 'BATTERY' | 'GAS'
  fuelPerTile            // consumption
  windPenalty            // multiplier on time when traveling against wind
  stabilityForCasting    // minimum cast quality penalty in chop (read by castTiming)
  upgradeSlots           // for trolling motor / fish finder / etc.
```
poiTravel.js cost calculation (conceptual pipeline)
```
1. baseTime  = edge.distanceTiles / boat.speedTilesPerMin
2. windCost  = baseTime × max(1, windAlignmentPenalty(edge.bearing, wind, boat.windPenalty))
3. shallowOK = edge.minDepth >= boat.shallowDraftMin     // else: route blocked
4. fuelCost  = edge.distanceTiles × boat.fuelPerTile     // 0 for oars
5. travelTime = baseTime × windCost
6. emit TRAVEL_STARTED { edge, eta: now + travelTime, fuelCost }
7. clock.schedule(travelTime, () => emit TRAVEL_ARRIVED ...)
```
Why the rowboat actually hurts (per your design intent)
`speedTilesPerMin` very low → big `baseTime`.
`windPenalty` high → wind compounds the slow speed instead of being a fixed mult.
`fuelType: 'NONE'` but **player-effort cost** could be modelled later as a stamina drain.
`shallowDraftMin` low (rowboats can go anywhere) → tradeoff is real: rowboats reach POIs that high-speed bass boats cannot.
This last point is the hidden gameplay depth: upgrading isn't strictly better. A flat-bottom skiff and a 250hp bass boat have different POI graphs available, not just different traverse speeds. poiGraph.js already supports this — edge.minDepth is part of the graph; the boat's shallowDraftMin filters which edges are traversable.
Recommendation
Boats are stat blocks in a new equipment/boats.js. poiTravel.js reads the active boat from state.hub.activeBoat, applies the pipeline above, and is otherwise unchanged. Boat shop is a hub menu that mutates state.hub.activeBoat after a wallet check.

4.3 Append to §8 KNOWN HAZARDS
```
- H-005 Mode Leak: Subsystems left subscribed to the bus across a mode change (e.g., AI scheduler still ticking in HUB) cause silent state drift and battery/clock waste. Mitigation: every subsystem must register a mount manifest with `onUnmount` that cancels all clock handles and bus subscriptions. Engine boot test asserts zero stray subscriptions after a HUB↔TOURNAMENT round-trip.
- H-006 Persistence Tampering / Cheat: Profile JSON on disk is a tempting save-edit target. Mitigation deferred, but flagged: design for a checksum / signed save file before v1.0.
- H-007 Audio Queue Backpressure: A flood of bus events (e.g., rapid sonar pings during fast travel) can swamp the TTS queue. Mitigation: `audio/ttsQueue.js` enforces priority + coalescing (drop duplicate `SONAR_PING` within window).
```
