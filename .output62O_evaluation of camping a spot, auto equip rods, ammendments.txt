Macro Design Evaluation
§1 — Holistic Design Critique
1.1 Distance Vectors & Rod Auto-Equip
Verdict: SOUND with one critical correction.
The rod-reach gating is a brilliant information mechanic — it makes the rod ladder (D-045: UL→XH) economically legible. Long rods stop being "bigger number" and become spatial access keys, exactly like boat draft (D-022) is a depth access key. Excellent symmetry.
However, "auto-equip" as worded would violate H-017a (Active Tackle Mutation Boundary). Once TOURNAMENT_ACTIVE is entered, set-membership of activeTackle.rods is FROZEN. Pulling a new rod from broader Hub inventory mid-tournament is forbidden by design.
Correction: rename and rescope to Rod Auto-Select: the cast pipeline picks the best-fit rod from the already-rigged `activeTackle.rods` set. A target whose distanceM exceeds every rigged rod's reach is then "out of range — re-rig at dock" (greyed out). This converts the mechanic into a pre-tournament loadout puzzle (D-069 Loadout Rigging gains real teeth) and strengthens the Hub economy: now the player must own AND bring a long rod.
1.2 Dynamic Lure Selection
Verdict: SOUND but audio-UI risk is real.
Strategically excellent — sonar data finally has a consumer decision attached, not just a TTS info-dump. Without this, finder data is read-only flavour.
H-017 is not actually violated — Dynamic Lure Selection selects from activeTackle.lures (the frozen set). Frozen Tackle (the SET) survives; "Frozen Lure Choice" (a single lure pre-bound to the rod) is what dies. That's the rule to repeal, and it's an implicit rule, never codified — easy demolition.
Audio-UI tedium risk: with 5–20 lures and a list-read of categories, color, weight, profile — players will rage-quit. Mitigation (mandatory):
**Smart pre-sort**: `targetSelector` ships a `recommendedLureId` field (best `M_lure↔species × M_lure↔poi` from D-059) so default = best match. Single-tap to accept.
**Categorical-only TTS**: announce category + tier + sweet-weight match boolean (3 tokens), not full schema. Mirrors D-065 finder-ping philosophy ("ping never blocks TTS").
**One-button override**: ARROW_UP/DOWN cycles through `activeTackle.lures` sorted by match score; SPACEBAR confirms. Identical UX shape to `targetSelector` — players already learned it.
1.3 Camping Loop
Verdict: SOUND — this is the single best idea in the proposal.
D-038 (Spook) and D-039 (Pressure) already encode the diminishing-returns math. They are currently underused because every cast restarts the loop from SCAN. Camping converts these from "background decay" into the central tactical tension: stay or move?
FSM impact is minimal. The cast FSM resets to IDLE on CAST_LANDED today. We add state.tournament.lastTarget, retained across CAST_LANDED, cleared on (a) new TARGET_LOCKED, (b) SCAN, (c) POI travel, (d) tournament end. castPipeline reads lastTarget as a Tap-1 commit shortcut.
D-012 (Wind Lock) survives intact — wind is re-sampled on every Tap 1, including camped recasts. Good.
1.4 Active Drift & GPS Trolling Motor
Verdict: SOUND — already half-built.
D-003 already declares "Active Oars vs Passive Trolling Motor." The proposal just adds a third tier: GPS Trolling Motor (auto-station-hold, premium battery drain). No invariant conflict.
D-012 ("Wind Lock") and D-011 ("POI-frame target anchor") are untouched — those govern cast-time anchoring, not navigation drift. The boat drifts pre-cast and post-splashdown freely; D-012 says only that wind sampled AT Tap 1 governs the in-flight ballistic. There is no "Static Anchor" invariant to solve — the user is misremembering. Confirmed by reading D-011/D-012 verbatim.
Economy boost: GPS Trolling Motor becomes a top-tier Hub purchase, tucked under hubShops.js boat-upgrade category. Excellent money sink for late-week wallets.

Conflict Resolution
H-017a (frozen activeTackle set) vs Rod Auto-Equip Rename to Rod Auto-Select; restrict to activeTackle.rods; out-of-range targets greyed in FISH_FINDER_RESULTS. H-017 preserved.
Implicit "one lure per cast" vs Dynamic Lure New FSM phase LURE_SELECT inserted between TARGET_LOCKED and ARMED. Selection from activeTackle.lures only — set still frozen. H-017 preserved.

D-012 "Static Anchor" myth No conflict exists. D-012 is cast-time wind sample, not boat anchor. Document this explicitly in new H-019.
Drift integration vs GPS Trolling Motor navigation.js already owns drift integration (H-001). GPS-tier motor sets driftStep(dt) = 0 while powered + battery > 0; consumes motor.batteryRemaining() per tick via existing motor.consume() contract. No new module.

Phase 11C — LURE_SELECT FSM phase (D-072, H-020)
**casting/castPipeline.js** — *single largest change*
- Add FSM state `LURE_SELECT` between `TARGET_LOCKED` consumption and `ARMED` entry.
- On `TARGET_LOCKED`:
  - Auto-select rod: `state.tournament.cast.activeRodId = equipment.bestRodForDistance(payload.distanceM).id`.
  - Build lure options from `equipment.listActiveLures()`, score each with `M_lure↔species × M_lure↔poi` (reuse D-059 weights), pick `recommendedLureId` = argmax.
  - Emit `LURE_OPTIONS { lures, recommendedLureId, atMs }`.
  - Enter `LURE_SELECT`; cursor starts on `recommendedLureId`.
- In `LURE_SELECT`: consume `INPUT_ARROW_UP/DOWN` to move cursor, `INPUT_SPACEBAR` to commit (emit `LURE_LOCKED { lureId, atMs }`, store as `state.tournament.cast.activeLureId`, transition to `ARMED`), ESC to cancel back to target list (emit no event; just unbind).
- **DO NOT** import from `equipment/` deeply — use the `equipment.js` Public API only (D-066).
Phase 11D — Camping retention (D-073)
**core/stateStore.js**
